## How the Registry System Works

### The Big Picture

```
src/registry/          →  build-registry.mts  →  shadcn build  →  /public/r/*.json
(component definitions)    (stage 1)              (stage 2)        (static JSON served via HTTP)
```

Users install components with:
```bash
pnpm dlx shadcn add https://yoursite.com/r/component-name.json
# or with namespace:
pnpm dlx shadcn add @your-namespace/component-name
```

---

### Step-by-Step to Replicate

#### 1. `components.json` — Register Your Namespace

The `registries` field maps a namespace to a URL pattern:

```json
{
  "registries": {
    "@ncdai": "https://chanhdai.com/r/{name}.json"
  }
}
```

This lets users run `pnpm dlx shadcn add @ncdai/wheel-picker` and the CLI resolves it to `https://chanhdai.com/r/wheel-picker.json`.

#### 2. `src/registry/` — Define Your Components

Each category has a `_registry.ts` file with item definitions:

```
src/registry/
├── index.ts                          # Aggregates all categories
├── components/
│   ├── _registry.ts                  # Component definitions array
│   └── wheel-picker/
│       └── wheel-picker.tsx          # Actual component code
├── hooks/
│   ├── _registry.ts
│   └── use-sound/
│       └── use-sound.ts
├── lib/
│   ├── _registry.ts
│   └── utils/
│       └── utils.ts
├── blocks/
│   ├── _registry.ts
│   └── login-01/
│       └── page.tsx
├── styles/
│   └── _registry.ts
└── examples/
    └── _registry.ts                  # Filtered out of public registry
```

A registry item definition looks like:

```typescript
// src/registry/components/_registry.ts
export const components: RegistryItem[] = [
  {
    name: "wheel-picker",
    type: "registry:component",
    title: "Wheel Picker",
    author: "ncdai <dai@chanhdai.com>",
    description: "iOS-like wheel picker...",
    dependencies: ["@ncdai/react-wheel-picker"],  // npm deps
    registryDependencies: [                         // other registry items
      getRegistryItemUrl("shimmering-text"),        // → https://chanhdai.com/r/shimmering-text.json
    ],
    files: [{
      path: "components/wheel-picker/wheel-picker.tsx",
      type: "registry:component",
    }],
    docs: "https://chanhdai.com/components/react-wheel-picker",
  },
]
```

The aggregator (`src/registry/index.ts`) combines all categories:

```typescript
export const registry = {
  name: "ncdai",
  homepage: "https://chanhdai.com/components",
  items: [...lib, ...hook, ...components, ...blocks, ...styles, ...examples],
}
```

#### 3. `src/scripts/build-registry.mts` — Stage 1 Build

This custom script:
- Reads all `_registry.ts` definitions
- Resolves relative file paths to absolute (`src/` prefix)
- Generates `src/__registry__/registry.autogenerated.json` (full registry)
- Generates `public/r/registry.json` (public copy)
- Generates `src/__registry__/index.tsx` (lazy-loaded component map for the website's preview UI)

#### 4. `shadcn build` — Stage 2 Build

The shadcn CLI reads `registry.autogenerated.json` and produces **individual JSON files** in `/public/r/`:

```bash
shadcn build ./src/__registry__/registry.autogenerated.json
```

Output: one JSON file per component with **inlined source code**:

```json
// /public/r/wheel-picker.json
{
  "$schema": "https://ui.shadcn.com/schema/registry-item.json",
  "name": "wheel-picker",
  "type": "registry:component",
  "dependencies": ["@ncdai/react-wheel-picker"],
  "files": [
    {
      "path": "src/registry/components/wheel-picker/wheel-picker.tsx",
      "content": "import \"@ncdai/react-wheel-picker/style.css\"\n... full code ...",
      "type": "registry:component"
    }
  ]
}
```

The `content` field contains the **entire file source** — this is what the CLI extracts when a user installs it.

#### 5. The Build Command

```json
// package.json
{
  "scripts": {
    "registry:build": "bun run ./src/scripts/build-registry.mts && shadcn build ./src/__registry__/registry.autogenerated.json"
  }
}
```

#### 6. Serving — Static Files, No API Route

The JSON files live in `/public/r/`. Next.js serves them as static assets. No API route, no server-side generation. When someone hits `https://chanhdai.com/r/wheel-picker.json`, it's just a static file.

#### 7. Helper Utilities

```typescript
// src/utils/registry.ts
export function getRegistryItemUrl(item: string) {
  return `https://chanhdai.com/r/${item}.json`
}

export function getRegistryItemNamespace(item: string) {
  return `@ncdai/${item}`
}
```

#### 8. Runtime Registry Access (for the website itself)

`src/lib/registry.ts` provides `getRegistryItem()` used by the docs/preview pages:
- LRU cache (5-min TTL)
- Reads component source from disk
- Fixes import paths (`@/registry/components/foo` → `@/components/foo`)
- Validates against shadcn schema

---

### Registry Item Types

| Type | Purpose | Install Target |
|---|---|---|
| `registry:component` | React components | `components/{name}` |
| `registry:hook` | Custom hooks | `hooks/{name}` |
| `registry:lib` | Utility functions | `lib/{name}` |
| `registry:block` | Full page layouts | `app/{target}` |
| `registry:style` | CSS/Tailwind utilities | styles |

### Minimum Viable Setup for Your Project

```
your-project/
├── components.json                    # with your namespace + URL
├── src/
│   ├── registry/
│   │   ├── index.ts                   # aggregator
│   │   ├── components/
│   │   │   ├── _registry.ts           # definitions
│   │   │   └── my-component/
│   │   │       └── my-component.tsx   # source
│   │   └── hooks/
│   │       ├── _registry.ts
│   │       └── use-something/
│   │           └── use-something.ts
│   ├── scripts/
│   │   └── build-registry.mts         # generates autogenerated JSON
│   └── __registry__/
│       └── registry.autogenerated.json # generated
├── public/
│   └── r/                             # generated static JSON
│       ├── registry.json
│       └── my-component.json
└── package.json                       # with "registry:build" script
```
